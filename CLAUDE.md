# smart-led — Claude instructions

## SD card longevity (ALWAYS follow these rules)

This server runs 24/7 on a Raspberry Pi with an SD card. SD cards die from writes. Every code change must be evaluated for disk write impact.

### What setup.sh already protects (do not undo)
- `/tmp` and `/var/log` → tmpfs (RAM)
- `noatime` on root filesystem
- journald `Storage=volatile` — logs never touch SD
- swap disabled
- `PYTHONDONTWRITEBYTECODE=1` in the service unit

### Rules for any new code
- **No logging to files.** Use `log.info()` / `log.error()` to stdout/stderr only — the service redirects these to `/dev/null` via systemd (`StandardOutput=null`).
- **No writing to disk in loops or per-frame.** Animation step functions run at 50 FPS. Never write to disk inside any animation loop, effect thread, or periodic timer.
- **No new persistent state files** without a hard cap and rotation. The only persistent write path is `ai_animations/` JSON, which is capped at 100 files (`_MAX_SAVED_ANIMATIONS` in `ai_animations.py`). If you add any new on-disk storage, add a matching cleanup/rotation mechanism.
- **No sqlite or database files** on the SD card without explicit justification.
- **Prefer `/run/` or `/tmp/` for ephemeral state** — both are tmpfs on this Pi.
- **Do not add swap**, memory-mapped files, or anything that spills RAM to disk.

### Audit checklist for new features
Before implementing any feature that involves storage, answer:
1. Does it write to disk? How often?
2. Is it bounded (fixed max size / rotation)?
3. Can it go to tmpfs (`/run/`, `/tmp/`) instead of a persistent path?
4. Does it write inside a loop or on a timer?

## Hardware
- Raspberry Pi 5 (4GB RAM)
- 300-LED WS2812B strip on GPIO 18
- AI animations generated by Gemini 2.5 Flash (Google AI Studio free tier — 500 RPD, 15 RPM)
- `gemini-2.5-pro` has near-zero free quota — do not set it as default
- `GEMINI_API_KEY` must be set in the systemd service environment (not shell exports — systemd has its own env)
